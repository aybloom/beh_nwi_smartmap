Built Environment Neighborhood Walkability Index SmartMap

2025-11-25

1. Introduction

This documentation is about preparation of three data sets to support creation of a walkability index for the local neighborhood around geocodes (See Rundle et al. 2019.). The downloaded and processed data is housed in the nwi_smartmap database hosted by uw csde.

Three data variables:

  1. Street intersection density
  2. Population density
  3. Business density

Street intersection density is a central component of walkability indices. Greater intersection density is positively correlated with “connectivity” and negatively correlated with block face length. That is, smaller block faces result in more intersections. Smaller block faces result in a greater ability to visit locations within a study area, and a larger “network buffer”, ceteris paribus.

We present an analytical framework and results for obtaining street intersections and subsequent intersection density within a Euclidean buffer for geocoded point locations using TIGER/Line shapefile data (for roads/intersections) and census tracts (for land area).

Population density was likewise developed from US Census data (American Community Survey 5 year estimates). A key feature of this processing is to remove water areas using National Hydrography Data (NHD), so that all density measures used only the land area as the denominator, rather than the area of a home-based buffer, which could include water.

Business density was developed using Data Axle (formerly Infogroup/InfoUSA) data.

This variant of the data preparation uses the SmartMap format (Hurvitz et al. 2014). The benefit of using the SmartMap format is that the GIS processing has been done ahead of time. This obviates the need to select a point, create a buffer, and then measure from different GIS layers to generate the three input variables. With the SmartMap format, one only needs to provide a set of points representing a cohort, and then to pick the three raster values for each point. The three variables are then Z-scored and summed to create a walkability index for the cohort.

2. Methods

2.1 Pseudocode

All data stored as geographies, EPSG 4269. Processing completely within R.

  1. create schemas/tables 1. tracts 2. water areas 3. edges 4. businesses

  2. land area (denominator) 1. get tracts and water areas for all counties in the selected year(s) using tigris 2. erase water features 3. convert to raster using a value of 1 (i.e., binary value indicating the cell represents land) 4. focal count of cells in 1 km radius (i.e., count of cells within buffer); multiply by cell area to get area of land in buffer

  3. intersections 1. get edge data for all counties $year 2. create intersections using start/end points from edges, snap points, count aggregate (i.e., number of unique points at the same XY for the same year) 3. Remove danglers (dead-end streets) 4. Recalculate aggregate endpoints with danglers removed. 5. limit intersections to ≥3 way 6. buffer intersections 7. convert buffers to raster where value is the number of “stacked” buffers at any cell location in the study area = count of intersections divide count of intersections raster by land area raster to get intersection density

  4. population density 1. get population data using tidycensus and join to water-erased tracts 2. using water-erased tracts with population counts, calculate count of persons per cell (new field in table) 3. focal sum of persons per cell, results in estimate of count of persons within 1 km buffer 4. divide count of persons raster by land area raster to get population density

  5. businesses 1. select set of walkable businesses by SIC code (in the Enclave) 2. buffer points (in Envlave) 3. convert buffers to raster where value is the number of “stacked” buffers at any cell location in the study area (in Enclave) 4. divide count of businesses raster by land area raster to get walkable business density (outside of Enclave)

2.3 Download Tract-level data

Function uses tigris to download tract data for set years.

2.3.1 Erase water from tracts

Water was erased from tracts using the US Census water data. note that water data is not available for years prior to 2011, so for tract years <2011, use 2011 water data

Pseudocode:

  1. test for intersection tract multipolygon with water multipolygon
  2. if there is no intersection, add tract to selected set
  3. if there is an intersection, perform st_difference() to obtain tract that is not intersecting with water; add to selected set

2.3.2 Population data

Population data were downlaoded using tidycensus.

2.3.3 Merge Land no water with population data

Tract-level population data was merged with tract area data (water-erased), and the population per raster cell is calcualted by dividing the tract population by the count of cells in a tract:

population / (st_area(geom_26910) / cell_area) as population_per_cell. The cell_area term in the current dataset is 10,000m, or the area of the cells (100m * 100m)

2.4 Download Edges/Line data

The county geoids were used to download TIGER/Line edges (which are available on a county-by-county basis). These were used to generate intersections. Note: TIGER/Line edges for 2007 must use a different format for downloading

2.4.1 Generating intersection points, buffering

Intersections were generated by pulling the start point and the end point of each TIGER/Line edge from edges matching the criteria for walkable streets, mtfcc ~* ’S.*’ AND mtfcc NOT IN (‘S1100’, ‘S1500’, ‘S1630’, ‘S1640’, ‘S1730’, ‘S1740’, ‘S1750’, ‘S1780’), see TIGER/Line Shapefiles Technical Documentation.

The start and end points were snapped to 0.1 based on examination of previous output in which edges at locations that appeared to have an intersection generated none. One could consider start- and/or end-points at the same location to be “stacked”, allowing for counting multiple points at the same XY location. To remove cul-de-sacs, we removed lines whose start- and end- points were stacked. We also removed danlgers, or lines who had no stacking on one end of the line. A location where only two edges meet would have two nodes, but a location where more than two unique edges meet would have at least three nodes (form unique edges). Thus, after isolating the end points, aggregating by location, excluding cul-de-sacs and dangers, and excluding aggregates with less than three nodes, we have a complete set of intersections.

All of the download formatting functions are run in a loop with the function below:

# download and add tract land, water, land with water erased, population variables and edges to database
f_add_data_smartwalk_db(conn = U, years=year_list, state_abbr_list = state_abbr_list)

2.5 Rasterize data

2.5.1 Check raster file availability for years / states

List of the raster files that have been created so far for each year.

Raster files should include: (1) land area, (2) population density, (3) intersection density, and (4) business density.

check_raster_files()

2.5.2 Rasterize and combine tract-level land data

Combine tract-level data for each state/year from the database and rasterize using 1 to indicate land:

  1. Pull all tract_no_water data for each state/year from the database
  2. Combine all tract_no_water data for each state/year
  3. Rasterize tract_no_water data using a value of 1 (i.e., binary value indicating the cell represents land).
  4. Generate a count of cells within 1km (focus processing)
  5. Convert to area in hectares by dividing by 100.

In the final land area raster file, each cell contains the total hectares per 1km buffer.

2.5.3 Rasterize and combine tract-level population density data

  1. Rasterize population per cell for each state/year.
  2. Pull all tract_no_water data for each state/year from the database
  3. Combine all tract_no_water data for each state/year
  4. Rasterize tract-level data using the population per cell variable (raw state density pop raster files).
  5. Generate the estimated count of persons within a 1km buffer by summing all cells within a 1km buffer (summed state density raster files)
  6. Divide the summed population counts by the land density raster file to generate the population density per hectare (hectare state density pop raster file).

In the final population density file, each cell contains the population density (total people per hectare) for a 1km buffer.

2.5.4 Rasterize state-level intersection density data

Using the database intersection data (>=3 nodes), rasterize data file and get intersection density. 1. Generate buffers for each intersection 2. Rasterize the sum of the stacked intersection buffers 3. Divide the sum of stacked intersection buffers by the the land area raster to get intersection density.

In the final intersection density file, each cell contains the intersection density (total intersections per hectare) for a 1km buffer.

Raster files for each dataset are created are created in a loop for each state / year in the provided state and year lists.

2.5.5 Rasterize state-level business density data

Businesses were handled within the database by creating a partitioned table where each partition represents a year of business data. This allowed using a single query to pull multiple years of data. A query was generated to select all walkable businesses based on a list of SIC codes. The walkable businesses from Data Axle business data were selected using SIC codes identified by Bethany Williams.

Each walkable business was buffered with a radius of 1 km.

The business point extract code was written on UWDC Enclave 1 (Where the Data Axle data are housed). 1. Import 2016 land density raster file to ensure consistent extent of generated business density raster file 2. Select walkable business from Data Axle database and filter by state, year, and SIC code. 3. Buffer business points by 1km 4. Generate a sum of stacked business points raster file 5. Export the business data raster file 6. Outside of the UWDC Enclave, divide the business data raster file by the land density file to get the walkable business density.

In the final business density file, each cell contains the total walkable businesses per hectare for a 1km buffer.

Note: use the code in the r_code_base folder while in the data enclave to create a raster file of businesses for a given year and state. The get_raster_stacks.r file creates and runs the function to create a raster.

To export the raw raster file, use the Data Enclave process to export from the data enclave and save the state_raster/business/{yr}/sum folder to run the function below.

2.6 Create a composite measure

The Composite walk measure is created by finding the year- and state- specific z-score of (1) businesses density, (2) population density, and (3) street network density and summing to create a composite measure. The function below calculates the z-score for each state/year and creates a raster files with the composite scores for a given year/state.

 state_fips <- fips_codes %>% filter(state %in% toupper(state_abbr_list))
for(yr in year_list){
  for(f in state_fips){

# create composite walk scores

                                              state_fips = f)
}}
